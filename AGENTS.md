# PawCircle Agent Guide

本文件用于指导 AI 助手在 `PawCircle` 仓库内协作，目标是：在不破坏现有风格与结构的前提下，快速完成前后端联调、页面开发与缺陷修复。

## 1. 项目概览

- 项目名称：`PawCircle`
- 结构：前后端分离
- 前端目录：`frontend/`
- 后端目录：`backend/`
- 前端技术栈：`Vue 3`、`TypeScript`、`Vite`、`Element Plus`
- 后端技术栈：`Node.js`、`Express`
- 业务方向：宠物社交、宠物交易、AI 对话、用户资料与好友互动

## 2. 基本协作规则

- 始终使用中文与用户沟通。
- 在修改代码前，先阅读相关文件，不凭空猜测实现。
- 优先做最小必要改动，不做无关重构。
- 保持现有命名、接口封装方式、组件组织方式一致。
- 不要为了“更优雅”而新增不必要的抽象层。
- 除非用户明确要求，否则不要新增依赖。
- 除非用户明确要求，否则不要改动 README、总结文档或 SQL 文件。
- 修改后如涉及已编辑文件，应检查明显报错、类型错误和语法问题。
- 进行改动时，要补充必要的说明


## 3. 目录职责

### 前端

- `frontend/src/views/`：页面级视图
- `frontend/src/components/`：可复用组件
- `frontend/src/api/`：接口请求封装与类型定义
- `frontend/src/utils/`：请求工具、事件总线等通用能力
- `frontend/src/router/`：路由配置

### 后端

- `backend/src/controllers/`：控制器逻辑
- `backend/src/routes/`：接口路由
- `backend/src/middleware/`：中间件
- `backend/src/config/`：配置项，如数据库配置
- `backend/src/utils/`：通用工具

## 4. 前端编码约定

- 优先沿用 `script setup` + `TypeScript` 风格。
- API 请求统一复用 `frontend/src/api/` 与 `frontend/src/utils/request.ts` 的既有方式。
- 页面提示优先使用 `ElMessage`，不要混用多套提示方案。
- 现有页面中较多使用：
  - `ref`、`reactive`、`computed`
  - `defineAsyncComponent`
  - `onMounted`、`watch`
- 样式优先延续现有写法：
  - 在 `.vue` 文件内使用局部样式
  - 尽量复用已有 CSS 变量
  - 不随意更换整体视觉语言
- 文案默认使用中文，风格自然、轻松，符合宠物社区产品气质
- **组件化思维**：设计与开发新功能或页面时，必须优先考虑是否可以将其设计并拆分成组件进行复用，避免大量代码堆砌在单一文件中，以提高代码的简洁度与可维护性。
- 组件优先延续现有写法：
  - 优先查看 `frontend/src/components` 里的组件是否可以复用
  - 其次查看 `frontend/src/views` 里各个文件里的 `components` 文件是否可以复用
  - 多个地方重复利用的组件放在 `frontend/src/components`
  - 当前页面内重复利用的真实 UI 组件（如卡片、弹窗等）放在此页面单独的 `components` 目录里
  - **切记**：子页面、子路由视图等非 UI 碎片的页面级组件，绝对**不要**放入 `components` 目录，应直接与父页面同级或放在对应的业务层级下。

## 5. 接口与数据约定

- 前端接口文件通常以 `xxxService` 命名导出函数，例如：`getPetsService`、`deletePetService`。
- 不要跳过现有 API 层直接在页面里手写请求，除非项目里该模块本来就是这样实现的。
- 新增接口时，优先放入对应业务文件，如：
  - AI 相关放在 `frontend/src/api/ai.ts`
  - 宠物相关放在 `frontend/src/api/pet.ts`
- 修改接口字段前，必须同时确认前后端字段是否一致。
- 若接口返回结构已有统一模式，优先保持兼容，不擅自改响应格式。

## 6. 页面开发偏好

- 页面以可用、清晰、直接为第一优先级。
- 不要引入与当前页面无关的大规模动画或复杂交互。
- 新增交互时，优先与现有页面风格保持一致。
- 表单页面重点关注：
  - loading 状态
  - 基本输入校验
  - 成功/失败提示
  - 提交后跳转或状态更新
- 列表页面重点关注：
  - 空状态
  - 加载状态
  - 分页、筛选、搜索是否与现有逻辑一致

## 7. 后端改动规则

- 后端优先沿用现有 `routes + controllers` 组织方式。
- 不要把大量业务逻辑直接塞进路由文件。
- 新增接口时，按现有模块拆分到对应 `controller` 和 `route`。
- 每次涉及数据库结构更改（如新增表、新增字段、修改字段类型等）或开发需要数据库配合的新功能时，**必须在回答中显式提供相应的 SQL 更改代码**，以便用户同步更新数据库。
- 若变更数据库字段或 SQL，除非用户明确要求，否则先不要主动改动 `.sql` 文件。
- 若前端报错来自后端字段缺失，先确认控制器返回结构，而不是先在前端写临时兼容逻辑。
- 前端新增页面或者功能后，要自动联通新功能的前后端以及数据库

## 8. 修改优先级

当用户提出需求时，优先按以下顺序思考：

1. 先定位相关视图 / API / 路由文件
2. 阅读现有实现方式
3. 在原有结构上最小改动
4. 如需新增代码，优先放到已有同类文件中
5. 完成后检查受影响文件是否有明显问题

## 9. 不建议做的事

- 不要无原因重命名大量变量或文件。
- 不要顺手大改样式体系。
- 不要把简单逻辑拆成过多工具函数。
- 不要在一次小任务里顺带修一堆无关问题。
- 不要为了兼容假设场景增加过多分支。
- 不要在没有确认的情况下修改数据库结构。
- 不要擅自修改页面的整体设计风格
- 不要过度夸赞或谄媚回答
- 不要擅自更改项目的依赖项，例如 `package.json` 等

## 10. 与用户协作时的输出风格

- 回答简洁直接，先说结果，再补充必要说明。
- 回答时保持绝对客观理性正确。
- 解释代码时，结合当前仓库路径说明，不空谈概念。
- 如果需要用户决策，给出清晰选项。
- 如果信息不足，先问关键问题，不盲改。
- 要一步步进行思考回答
- 遇到不清楚的要求，不要猜测，要询问用户得到清楚答案再执行
- 用户提供要求后，优先使用 `plan` 模式给出方案，用户确认后再执行

## 11. 适合本项目的默认工作方式

- 用户提到页面问题时，优先查看 `frontend/src/views/` 下对应页面。
- 用户提到接口问题时，优先同时查看：
  - `frontend/src/api/` 中对应文件
  - `backend/src/routes/` 与 `backend/src/controllers/` 中对应模块
- 用户提到登录、注册、用户资料时，优先检查 `auth`、`user` 相关代码。
- 用户提到宠物交易时，优先检查 `trading`、`listing`、`social` 相关代码。
- 用户提到 AI 聊天时，优先检查 `frontend/src/api/ai.ts` 与 `backend/src/controllers/aiController.js`。

## 12. 文件放置建议

若需要新增文件，优先遵循：

- 新页面/子路由视图：直接放在对应的业务模块目录 `frontend/src/views/<module>/` 下（不要放进 `components`）
- 新组件：
  - 仅当前页面用的 UI 碎片组件（如特定卡片）：放在对应页面的 `components/` 
  - 全局通用组件：放在 `frontend/src/components/`
- 新接口：放在 `frontend/src/api/` 下已有业务文件中；仅在必要时新增业务 API 文件
- 新后端接口：同时补齐 `routes` 与 `controllers`

## 13. 最终原则

这个仓库更适合“顺着现有写法继续做”，而不是“按理想化工程方案重塑一遍”。

能在现有结构中解决，就不要额外抽象；
能小改完成，就不要大改；
能和当前风格一致，就不要另起一套。
